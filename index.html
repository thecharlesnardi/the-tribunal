The Tribunal

MMCS Automation System

⚙️

The Tribunal automates the Multi-Model Consultation System (MMCS) — a methodology that queries multiple AI models as complementary cognitive sensors, synthesizes their diverse perspectives, and stress-tests findings through adversarial verification. Twenty files. Seven agents. Nine models. One question at a time.

The Core Idea

The foundational insight: architecturally diverse AI models function more like different senses than like multiple pairs of eyes. Models share significant training data overlap — their “votes” are correlated, not independent. Different model architectures don’t just produce different answers — they perceive different aspects of the problem space entirely.

Model Archetype Analogous Sense What It Perceives
Constitutional AI (Claude) Proprioception Internal consistency, structural coherence — “does this hold together?”
Academic-heavy (Gemini) Vision Theoretical landscape, formal relationships — “what’s the big picture?”
Real-time integrated (Grok) Hearing Current discourse, practitioner chatter — “what are people actually saying?”
Search-integrated (Perplexity) Touch Surface coverage, citation density — “what sources exist?”
Tool-use native (Kimi) Taste Operational feasibility, implementation texture — “how would you actually build this?”
RLHF-optimized (ChatGPT) Smell User satisfaction patterns, completeness signals — “does this feel complete?”
“You don’t average what your eyes see with what your ears hear. You synthesize them into a unified model of reality that neither sense could produce alone. That’s the human’s role in MMCS.”
🌬️

The Breathing Pattern. The methodology isn’t a linear funnel — it’s rhythmic oscillation between expansion and contraction. DIVERGE is the “full inhale” — expanding the solution space across architecturally diverse models. COMPARE is the “exhale” — contracting the space by identifying what’s signal and what’s noise. Each cycle is one breath. Complex problems require multiple breaths until the system reaches equilibrium.

The Pipeline

Seven phases. One breath per cycle.

Phase Breath What Happens
SEED — The raw human intent. A question, a problem, a half-formed idea. Deliberately unsophisticated. The barrier to entry is zero. Record verbatim. If --recurse, the SEED is the gap/conflict from the parent cycle.
AMPLIFY Inhale begins AI transforms the seed into a sophisticated, well-structured research prompt covering 4–6 research dimensions chosen to produce maximum divergence across model architectures. Human reviews and approves before DIVERGE begins.
DIVERGE Full inhale The identical amplified prompt is sent to all selected models. No per-model customization — identical input, divergent output. CLI models run in parallel via ask-models.sh; browser models run sequentially via the dispatcher agent; Claude answers natively. Minimum 3 successful models required.
COMPARE Exhale begins Structured synthesis by the Synthesizer (Opus). Surfaces CONSENSUS, CONFLICTS, GAPS, and COST FLAGS. Every point attributed to specific models — no anonymous claims.
DECIDE Exhale completes The human Captain reviews the synthesis and chooses: proceed to CONVERGE, run CRUCIBLE, RECURSE into a sub-cycle, or adjust scope and re-run DIVERGE.
CRUCIBLE Stress test Adversarial verification of consensus findings. Generator defends, Attacker attacks (with WebSearch), Adjudicator renders independent verdicts. Optional at Standard, mandatory at Forensic+.
CONVERGE Final exhale Ledger Clerk assembles the Claim Ledger (structured inventory with confidence tiers) and the Final Report (narrative synthesis with executive summary, key findings, recommendations).

The Model Portfolio

Nine models. Three dispatch methods. Each model perceives the problem space through a different cognitive lens — that architectural diversity is the system’s primary information source.

Model Dispatch Cognitive Style Best Deployed For
Claude Native Epistemic guardian. Extensive literature grounding, honest uncertainty flagging. Resists sycophancy. Synthesis, evaluation, structural analysis, anything where intellectual honesty matters more than speed.
Gemini CLI Theoretical architect. Connects to formal research traditions, invokes established frameworks. Theoretical grounding, literature review, formal framework development.
Codex CLI Code-native reasoning. Technical implementation focus. Technical specifications, code generation, implementation planning.
ChatGPT Browser Comprehensive satisfier. Thorough, aimed at completeness. Generates detail that feels authoritative. Initial broad coverage, well-documented topics, content generation.
Grok Browser Fast practitioner. Specific tooling names, concrete thresholds, actionable heuristics. Current landscape assessment, practical recommendations, tooling selection.
Perplexity Browser Citation machine. Massive source coverage, numbered references, broad survey approach. Literature discovery, source verification, broad landscape mapping.
DeepSeek Browser Implementation architect. Clean structures, diagrams, test frameworks, operational focus. Implementation planning, technical specifications, system architecture.
Kimi Browser Operational innovator. Surfaces practical problems others gloss over. Names the actual hard problems. Identifying implementation challenges, edge cases, operational problems that polished models smooth over.
Genspark Browser Multi-agent search. Generates comprehensive sparkpages via distributed search. Comprehensive research synthesis, multi-source aggregation.
📡

Three dispatch methods. Native — Claude answers directly in the orchestrating session, no external call needed. CLI — Gemini and Codex are dispatched in parallel via the ask-models.sh bash script. Browser — all remaining models are queried sequentially through Chrome automation via the mmcs-diverge-dispatcher agent.

Rigor Tiers

Four tiers of rigor scaled to the stakes of being wrong. The selection heuristic: “If this finding is wrong, what happens?”

Tier Models CRUCIBLE Pre-reg Time Use When
Lite 3–5 None None 30–45 min Nothing much happens if wrong. Everyday decisions, quick comparisons, initial exploration.
Standard 5–7 Optional Recommended 60–90 min Wasted effort or wrong direction. Most research, strategic decisions, content development.
Forensic 7–9 Mandatory (3-role) Mandatory 2–3 hr Financial loss, reputation damage, or legal exposure. High-stakes decisions that may be challenged.
Academic 7–9 Mandatory (3-role) Mandatory 4+ hr Public record, peer scrutiny, or foundational strategy. Publishable research, methodology development.

Forensic adds hypothesis stacking (three framing variants per query) and source genealogy in the Claim Ledger. Academic adds literature integration, calibration tracking with Brier scores, methodology documentation for replication, and a formal limitations section.

The Agent Architecture

Seven specialized agents. Each writes results to disk and returns only a one-line confirmation — the context overflow prevention pattern, learned the hard way in earlier multi-agent work.

Agent Role Model What It Does
mmcs-amplifier Amplifier (AMPLIFY) Sonnet Transforms raw SEED question into structured research prompt. Does NOT answer the question — only structures it.
mmcs-diverge-dispatcher Browser Dispatcher (DIVERGE) Sonnet Browser automation agent that queries one AI model per invocation via Claude-in-Chrome MCP tools.
mmcs-synthesizer Synthesizer (COMPARE) Opus Reads all model responses and produces CONSENSUS / CONFLICTS / GAPS / COST FLAGS synthesis with model attributions.
mmcs-crucible-generator The Advocate (CRUCIBLE) Sonnet Defends synthesis claims with the strongest possible evidence and reasoning — builds the prosecution’s case FOR each finding.
mmcs-crucible-attacker The Adversary (CRUCIBLE) Opus Systematically attacks every claim. ONLY agent with WebSearch — can find 2026 evidence not in training data. Uses 5-part CRUCIBLE protocol.
mmcs-crucible-adjudicator The Judge (CRUCIBLE) Opus Reads Generator and Attacker briefs, renders independent verdict per claim. Does not side with either party. Truth is weighed, not negotiated.
mmcs-ledger-clerk Ledger Clerk (CONVERGE) Sonnet Assembles the final Claim Ledger and narrative report from synthesis + CRUCIBLE verdicts. Updates persistent tracking files.
💡

Why Opus for three roles? The model hierarchy is intentional. Opus handles the three judgment-critical roles: synthesizer, attacker, adjudicator. Sonnet handles everything else: amplifier, dispatcher, generator, clerk. Save the expensive model for where judgment quality matters most.

The CRUCIBLE

Named for metallurgy: a crucible holds metal at extreme heat to test its purity. What survives the crucible is trustworthy. CRUCIBLE is an adversarial child cycle spawned within MMCS when the stakes of being wrong exceed the cost of verification. It’s not a new research cycle — it’s a stress test on existing findings.

Trigger Conditions

  1. Unanimous convergence on a non-obvious claim — agreement itself is suspicious.
  2. Source genealogy concern — all models trace to the same upstream research.
  3. Action-forcing findings — irreversible decisions, resource commitments, public claims.
  4. “Too clean” results — real problems have friction; smooth synthesis means something was smoothed over.
  5. Captain’s intuition — the human operator smells something off.
🛡️

The Advocate

Generator · Sonnet

Defends each finding with the strongest possible evidence and reasoning. Builds the prosecution’s case FOR each claim — model support, evidence base, logical chain, anticipated objections.

“The stronger your defense, the more meaningful the Attacker’s eventual test will be.”

⚔️

The Adversary

Attacker · Opus

Systematically attacks every claim using the 5-part CRUCIBLE protocol: methodological critique, contradicting evidence, boundary conditions, source genealogy, steel man counter-case. The ONLY agent with WebSearch — finds 2026 evidence that contradicts training-data claims.

“Don’t be fair. Be thorough. Fairness is the Adjudicator’s job.”

⚖️

The Judge

Adjudicator · Opus

Reads both briefs and renders independent judgment on each claim. Does not automatically side with either party. Verdicts: Withstood, Damaged, Fatal, or Transformed.

“TRANSFORMED is often the right verdict. Real findings frequently need refinement, not destruction.”

The Claim Ledger

The primary output document — not a narrative report, but a structured inventory of what you know and how well you know it. Each claim is tagged with a confidence tier, supporting evidence, CRUCIBLE testing notes, and recommended actions.

Tier Label Criteria Appropriate Action
1 VERIFIED Unanimous model convergence + survived CRUCIBLE + independent source confirmation. Build strategy on this. Defend publicly.
2 PROBABLE Strong convergence (4+ models) + no significant conflicts + plausible mechanism. Act on this with monitoring. Flag if challenged.
3 PLAUSIBLE Majority convergence OR strong single-model finding with theoretical backing. Use as working assumption. Seek confirmation before committing resources.
4 SPECULATIVE Minority finding, single-model insight, or unresolved conflict. Interesting lead. Investigate further before acting. Do not present as established.
5 CONTESTED Active conflict between models with strong arguments on both sides. Do not act on this without resolution. Present both sides.

CRUCIBLE verdict effects on confidence tiers: Withstood → maintain or raise confidence. Damaged → lower by one tier. Fatal → remove from ledger (note removal in appendix). Transformed → replace with transformed version, reassess tier.

The Spellbook Spell

The Tribunal is the fifth spell in the Spellbook — a reusable agentic prompt template that chains the full MMCS framework into five self-directing movements.

Movement MMCS Phases What Happens Agent(s)
1. The Seeding SEED + AMPLIFY Question captured, structured into research prompt. At Standard+, complete pre-registration before seeing any model output. mmcs-amplifier (Sonnet)
2. The Summoning DIVERGE Prompt dispatched to model portfolio. CLI models run in parallel via bash; browser models run sequentially via Chrome automation; Claude answers natively. mmcs-diverge-dispatcher (Sonnet) + ask-models.sh
3. The Weighing COMPARE + DECIDE Structured synthesis produced. Captain decides: proceed, enter Crucible, recurse, or adjust scope. mmcs-synthesizer (Opus) + human
4. The Crucible CRUCIBLE Three-role adversarial stress test. Generator defends, Attacker attacks with WebSearch, Adjudicator renders verdicts. Generator (Sonnet) → Attacker (Opus) → Adjudicator (Opus)
5. The Verdict CONVERGE Claim Ledger assembled with confidence tiers. Final narrative report written with executive summary, key findings, and recommendations. mmcs-ledger-clerk (Sonnet)

Variations

  • Quick Tribunal (Lite) — Movements 1–3, skip 4, jump to 5. Three to five models. No CRUCIBLE, no pre-registration. Best for everyday questions where you want multi-model perspective without full rigor.
  • Full Tribunal (Standard) — All 5 movements, 5–7 models. Pre-registration recommended. CRUCIBLE optional but encouraged. The default for most research.
  • Forensic Tribunal — 7–9 models with hypothesis stacking. Pre-registration mandatory. Three-role CRUCIBLE mandatory. Source genealogy in the Claim Ledger. For decisions that may face external scrutiny.
  • Crucible Only — Start from an existing synthesis. Skip Movements 1–3. Run only The Crucible (4) and The Verdict (5). For stress-testing findings you already have.
  • Recurse — Loop from Movement 3 back to Movement 1 with a gap as the new SEED. Happens naturally when the Captain chooses RECURSE at The Weighing.

How to Use It

🚀
/mmcs [question] --tier [lite|standard|forensic|academic]

Options:
  --models N              Override model count
  --recurse               Child cycle — prompt for gap as SEED
  --crucible-only          Start from existing synthesis path
